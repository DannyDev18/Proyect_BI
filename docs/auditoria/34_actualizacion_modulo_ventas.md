## Verificación manual con usuario seed de ventas (Fase 0, punto 1)

No se ejecutó contra un servidor corriendo dentro de esta sesión (auditoría de código estático); la verificación de las 3 rutas de fuga (H-V2) se hace ahora automáticamente vía tests de integración nuevos (`test_ventas_actualizacion.py`) que consultan con el usuario seed `ventas_gye@empresa.com` (codven `102`) un `cliente_id` fuera de su cartera y confirman `403`.

## Acciones aplicadas (Fase 1)

1. **H-V1:** se retira `GlobalBranchSelector` (componente ahora sin ningún otro consumidor, eliminado) y el estado `sucursal`/`setSucursal` muerto. El subtítulo del dashboard muestra la sucursal real que el backend ya aplica: "Consolidado Global" para gerencia/administrador, la sucursal propia para `ventas`.
2. **H-V2:** `PredictionService.get_churn_risk`/`get_product_recommendations`/`get_customer_segment` ganan un parámetro `codven_restriccion: str | None`; si se pasa, verifican vía `CatalogRepository.cliente_pertenece_a_vendedor` (nueva consulta) que el cliente tenga al menos una venta histórica con ese vendedor, lanzando `PermissionDeniedError` (403) si no. `sales.py` resuelve la restricción por rol (`None` para gerencia/administrador, `id_vendedor_origen` para `ventas`, mismo criterio que `resolve_sucursal_filter`). `Cartera360Service.get_detalle_cliente` ahora exige y aplica siempre el `codven` del vendedor autenticado (este módulo es self-scope sin excepción por diseño, RN-V3).
3. **H-V3:** `GET /gerencia/goals/periods` amplía su RBAC a `["gerencia", "administrador", "ventas"]`. `GET /analytics/ventas/goals` gana `anio`/`mes` opcionales (`AnalyticsService.get_sales_kpis` los propaga en vez de siempre usar `get_latest_period()`); el frontend agrega un selector de período poblado con `usePeriods()`.
4. **H-V4:** sin cambios de código (re-verificación limpia).
5. **H-V5:** `Cartera360Repository.log_gestion` deduplica un submit idéntico (mismo `usuario_id`/`cliente_sk`/`evento`) dentro de `CARTERA360_DEDUPE_DOBLE_CLICK_SEGUNDOS` (default 10s, configurable) devolviendo el registro ya existente en vez de insertar uno nuevo.
6. **H-V7:** nuevo `settings.CHURN_UMBRAL_RIESGO_ALTO` (default 0.5, preserva el comportamiento previo) reemplaza el literal `0.5` en `get_churn_risk`/`get_churn_risk_batch`.
7. **H-V8:** `query_meta` ya no filtra por la columna inexistente `m.sucursal` -- restringe las metas a los vendedores que efectivamente vendieron en la sucursal consultada durante el período (subconsulta contra `fact_ventas_detalle`, mismo criterio que ya usaba `query_actual`). `query_ranking` gana el mismo `JOIN`/filtro de sucursal que `query_actual`, para no mostrar el ranking global cuando hay un filtro de sucursal activo.

**Reglas transversales respetadas:** routers thin (la resolución de `codven_restriccion` es una función de una línea, no lógica de negocio); excepción de dominio `PermissionDeniedError` (no `HTTPException`) para la fuga de RLS; degradación con gracia intacta en `prediction_service` (los `try/except` existentes no se tocaron); contrato `association` v0.2.0 (`score`) sin renegociar.
