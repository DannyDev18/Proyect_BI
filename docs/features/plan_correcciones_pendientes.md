## 2. Hardening de seguridad (requiere decisión del usuario — diferido explícitamente en auditoría 36)

| # | Ítem | Decisión pendiente |
|---|---|---|
| S-1 | **`docs/credenciales_sistema.md` versionado en el repo** con credenciales visibles. | Mover el contenido a un gestor de secretos / `.env` no versionado y eliminar el archivo en un commit. **Purga del historial git** (`git filter-repo`) es irreversible y reescribe hashes — decidir si el repo es compartido antes de hacerlo. Rotar las credenciales expuestas independientemente de la purga. |
| S-2 | ~~CORS `"*"` por defecto.~~ | **Resuelto** (2026-07-15): el default de código sigue en `"*"` (no se rompe el flujo de dev), pero `validar_configuracion()` en `backend/app/core/config.py` ahora trata `CORS_ORIGINS == ["*"]` igual que `JWT_SECRET`/`POSTGRES_PASSWORD` inseguros — con `ENV=production` aborta el arranque (`ValueError`) si no se definió explícitamente; en dev/test solo un `WARNING`. `.env.example` corregido: documentaba `BACKEND_CORS_ORIGINS` en formato JSON, una variable que el código nunca lee (`config.py` lee `CORS_ORIGINS` como string separado por comas) — quedaba muerta. Verificado con test manual de las 3 combinaciones (dev+`*`, prod+`*`→falla, prod+explícito→pasa) y la suite completa (229 tests) en verde. |
| S-3 | ~~Rate-limiting del login: no existe.~~ | **Resuelto** (2026-07-15): `slowapi` (nueva dependencia en `backend/requirements.txt`) aplicado a `POST /auth/login` vía `@limiter.limit(settings.AUTH_LOGIN_RATE_LIMIT)` (default `"5/minute"`, configurable por env). El `Limiter` vive en `backend/app/core/rate_limit.py` (módulo propio para evitar el import circular `main.py` → `api.py` → `auth.py` → `main.py`). Contador en memoria por IP (`get_remote_address`), suficiente para el único proceso de este despliegue — sin Redis en la arquitectura. Test nuevo `test_login_rate_limit_bloquea_tras_N_intentos` en `backend/tests/integration/test_auth_endpoints.py` (5 intentos fallidos OK, 6to → 429). Verificado además con rebuild real de la imagen Docker (`slowapi` no estaba en la imagen anterior) y arranque limpio del contenedor `bi_backend` con los 6 modelos ML cargados. Lockout temporal por cuenta (no solo por IP) queda fuera de este alcance — mismo mecanismo cubre el caso de uso principal (fuerza bruta desde una IP). |
| S-4 | ~~Sin refresh token ni revocación.~~ | Decidir si el proyecto (tesis) lo necesita; si sí: refresh tokens cortos + blacklist en memoria/BD. | **Evaluado, no implementado** (2026-07-15): `backend/app/core/security.py` confirma un único JWT de 8h sin infraestructura de refresh/blacklist. Implementar el patrón completo (token corto + refresh + revocación) exige: un segundo tipo de token, endpoint `/auth/refresh`, y una blacklist persistida (no hay Redis en la arquitectura -- CLAUDE.md §Tecnologías -- así que sería una tabla `public.*` nueva con limpieza periódica). Es un cambio de arquitectura de autenticación, no una corrección puntual, y el riesgo principal que resolvería (usuario desactivado sigue con sesión válida) ya está mitigado por `es_activo` verificado en cada request (auditoría 36 H7) -- el JWT vigente de un usuario desactivado deja de poder usarse igual. Para el alcance de un proyecto de tesis, no se justifica sin un requisito explícito que lo pida; queda diferido hasta que exista esa necesidad concreta. |
